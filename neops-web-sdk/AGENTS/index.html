    neops-web-sdk

    Angular monorepo producing publishable @zebbra/ NPM library packages that provide the shared foundation for all neops web frontends. Includes UI components built on IBM Carbon Design System, GraphQL-backed domain services, JSON schema-driven form rendering, file storage integration, and shared build/deployment artifacts. Also contains a demo application (port 4201) serving as a living integration test.

    Tech: Angular 20, TypeScript 5.8, Carbon Design System, Apollo GraphQL 4, JSON Forms, Leaflet, ng-packagr, Karma/Jasmine, SCSS, Node 20.19

    Development

    nvm use                 # Node 20.19
npm install             # also runs postinstall -> build-schematics
npm start               # demo app at http://localhost:4201
npm test                # Karma/Jasmine unit tests
npm run lint            # ESLint (8GB memory limit; prelint runs check:icons)
npm run format:fix      # Prettier
make libraries          # build all libraries in dependency order
make scan-bad-imports   # check for incorrect cross-library imports

    Backend expected at http://localhost:8000/graphql during local dev.

    Packages

    Seven libraries live under projects/zebbra/ and must be built in the order make libraries uses (it is the dependency order; ngx-neops-icons is first):

    # Package NPM Name Purpose Key Exports
    1 ngx-neops-icons @zebbra/ngx-neops-icons Carbon icon resolution + preloading preloadIcons(), icon renderer component
    2 ngx-jsonforms-carbon @zebbra/ngx-jsonforms-carbon JSON Forms renderers styled with Carbon Design System Text, textarea, number, autocomplete, date, checkbox, toggle, time, range controls; vertical, horizontal, group, array, categorization layouts
    3 ngx-neops-app-services @zebbra/ngx-neops-app-services Cross-cutting application services AlertService, DialogService, NotificationsService, DownloadService, AppUpdateService, HealthCheckService, AutoRefreshService
    4 ngx-neops-client @zebbra/ngx-neops-client Domain services and GraphQL API client AuthService, AuthGuard, GraphQLModule, ConfigService, ContextService, entity services (Device, Interface, Client, Group, Task, Execution, Location, Scope, Topology, Report), ElementSearch<T>
    5 ngx-neops-app-components @zebbra/ngx-neops-app-components UI component library GenericTableComponent, DynamicTableConfigService, dashboard cards (many types), HeaderComponent, TaskCockpitComponent, NetworkDiagramComponent, MultiDashboardComponent, EditorComponent (Monaco)
    6 ngx-neops-sdk-artifacts @zebbra/ngx-neops-sdk-artifacts Shared build config, webpack, nginx, styles Custom webpack config (url-loader, raw-loader), nginx SPA config, Docker entrypoint, IBM Plex fonts, SCSS base styles

    The actual class names and exact public surface are authoritative in each library’s public-api.ts — re-list sparingly here, since these tables drift.

    Verification

    npm run lint && npm test && make scan-bad-imports

    Conventions

    • Default branch is develop. Branch from develop for all changes.
    • Libraries MUST NOT import from each other using relative paths – only via @zebbra/ package names (enforced by make scan-bad-imports)
    • All libraries use ng-packagr for building
    • Each library has a public-api.ts defining its public surface
    • Component prefixes: app (demo app), neops (client/components/icons libraries), lib (jsonforms/artifacts/storage)
    • All entity services follow the ElementSearch<T> interface pattern
    • Version placeholder 0.0.0-placeholder in all library package.json files – CI replaces via sed
    • SCSS is the default style language across all projects
    • Carbon v11 cds-- prefix is dominant (~240 uses); v10 bx-- is legacy/minimal (~15 uses)

    Code style

    tsconfig.json is strict (strict, noImplicitOverride, noPropertyAccessFromIndexSignature, noFallthroughCasesInSwitch, strictTemplates) and .eslintrc.json enforces most rules. The non-obvious conventions:

    • Standalone components by default (standalone: true). Module wiring in ngx-neops-app-components.module.ts exists for legacy reasons — prefer not to add to it.
    • Angular control flow (@if / @for / @switch), not *ngIf / *ngFor. Older templates are mid-migration; write new ones entirely in the new syntax.
    • inject() over constructor params for new code.
    • *.spec.ts next to the source file. Karma + Jasmine — no Jest.
    • Index-signature properties need bracket access (obj['title'], not obj.title) under noPropertyAccessFromIndexSignature — hits commonly when typing JSON-shape data.
    • Property selectors use the neops- prefix (selector: 'neops-execution-panel').
    • GraphQL types come from @zebbra/neops-angular-graphql-client. Wrap them in Neops* classes in ngx-neops-client/src/lib/neops-api/model/ when adding behavior (see NeopsExecution.ts’s mergeFrom).
    • Commit messages: descriptive prose, no enforced prefix. Match recent git log.

    Gotchas & Boundaries

    • NEVER build libraries out of order – dependencies between them will cause compilation failures. Use make libraries
    • NEVER modify the custom webpack config without understanding that ALL library builds, tests, AND consumer projects (neops-web-client) depend on it
    • ALWAYS run make scan-bad-imports after changing library imports
    • The postinstall script builds schematics – npm install will fail if the schematics build fails
    • npm overrides are extensive to force compatible peer deps for Angular 20
    • Lint requires NODE_OPTIONS=--max-old-space-size=8192 (configured in package.json script)
    • The Docker build sets NODE_OPTIONS=--max_old_space_size=4096 to prevent OOM (Dockerfile, not the local npm run build)
    • Token is stored in localStorage with the Bearer prefix already included (auth.service.ts); the access token auto-refreshes every 2 minutes

    Code-level gotchas (these have bitten contributors)

    • Dev server port is 4201, not 4200. Configured in angular.json under the serve options.
    • Monorepo build order matters. The dev app and ngx-neops-app-components import via file:projects/zebbra/<project> references and consume the dist/zebbra/<project>/ output. If you change ngx-neops-client, rebuild it before building app-components or running tests that import the new API, or the type checker sees a stale .d.ts.
    • fetchPolicy: 'network-only' still writes results into the Apollo normalized cache even though it bypasses it for reads, so other observers of the same query get re-notified on every poll. When you don’t want cache side-effects, fetch once with firstValueFrom(gql.fetch({...})) instead of watch().
    • exhaustMap propagates inner errors and tears down the outer timer(...). Wrap the inner Observable with catchError(... -> EMPTY) if polling should survive transient failures (see NeopsExecutionService.fetchExecution).
    • AuthGuard protects most app routes (/network/*, /tasks, /executions/:id, …). The /integration/* demo routes are unauthenticated.
    • Carbon combo-box event payload shape varies by handler. Single-select emits a ListItem, multi-select emits Array<ListItem>, and some versions wrap with { item: X }. Read defensively: event?.item ?? event.
    • Carbon listItems carry their selected flag inside the array element, not via two-way binding. Mutate the item, then push a fresh array reference to trigger change detection.
    • JSON Forms can return colorMapping (and other complex fields) as a parsed object OR a JSON string depending on the renderer path. Always normalize with a parse-or-passthrough helper before dereferencing.
    • generic-table opens a row-detail modal on double-click by default. For read-only tables set rowModalDisabled: true on the NeopsTableConfiguration.
    • Monaco needs an explicit editor.layout() call after the host resizes. EditorComponent uses ResizeObserver + double requestAnimationFrame; reuse that pattern if you embed Monaco elsewhere.
    • schema.graphql at the repo root is a captured introspection dump for grepping field names — not the source of truth. The generated @zebbra/neops-angular-graphql-client is. Do not edit it.

    Ecosystem Context

    • Consumed by: neops-web-client (and other neops frontends) via the published @zebbra/ packages
    • Depends on: @zebbra/neops-angular-graphql-client (code-generated GraphQL ops), @zebbra/neops-storage-angular-client (generated storage API client)
    • Depends on at runtime: neops-core (GraphQL API), neops-storage (file API)
    • Libraries published to GitHub Packages NPM registry (https://npm.pkg.github.com under @zebbra scope)

    Architecture

    Monorepo layout: library projects under projects/zebbra/, one demo app at src/, schematics at neops-schematics/.

    State management: No NgRx. RxJS BehaviorSubject patterns throughout. ContextService holds app-wide state (current scope, query filters, task context). AuthService manages auth state.

    Data flow: Pages create ElementSearch<T> instances -> GraphQL queries via Apollo -> results populate GenericTable or dashboard cards. Query filters synced to URL params via ContextService.

    Auth: Dual mode – GraphQL username/password and OIDC (Keycloak). JWT in localStorage. OIDC configurations are loaded from the backend at startup via AuthService.getOidcConfigurations() and applied through angular-auth-oidc-client.

    Search: Uses liqe for Lucene-like query parsing in search autocomplete/filters.

    Patterns worth knowing before changing things

    • Race-guard for async work whose results may arrive late: RequestVersionGuard (ngx-neops-client/common/). Start a token at the beginning of an async flow, check guard.isStale(token) before mutating state at any await boundary, and call guard.invalidate() in ngOnDestroy. Used by execution-panel and execution-details-shell.
    • Wrapper merge for polling refreshes: NeopsExecution.mergeFrom() preserves fields a smaller follow-up query did not return. Reuse this if you add another wrapper that needs partial refreshes.
    • Element-search for paginated lists: most list-backed components compose ElementSearchArrayResult rather than driving GraphQL directly (ngx-neops-client/element-search/).
    • Card config form adapters round-trip a different editor shape than the stored shape at the configurator boundary (see chart-color-mapping-form-adapter.ts).

    Testing

    • npm run test:ci runs all libraries’ Karma configs via the Angular CLI (ChromeHeadlessCI).
    • Scope to one library: npx ng test @zebbra/<project> --browsers=ChromeHeadlessCI --watch=false. The @zebbra/ prefix matters. Each karma.conf.js binds localhost:9876, so parallel runs collide with EADDRINUSE — serialize them or pass --port.
    • Stub generated *GQL services with spies in TestBed.providers (or use ApolloTestingModule).
    • For timer(...)/interval(...) pipelines, use fakeAsync + tick(ms) + flush().
    • Monaco-based components need the shared shim at src/testing/monaco-loader-test-shim.ts.
    • singleRun is false in the karma config but --watch=false overrides it. If a TS compile error stops karma from loading any test file, the runner can hang instead of exiting — kill it and fix the compile error.

    When to ask a human

    Schema/GraphQL-type changes (they come from a separate repo), routing/AuthGuard changes, new Neops* wrapper classes, reordering any library’s published public-api.ts, or refactors that require backend behavior changes — all affect external consumers. Surface them rather than deciding solo.
