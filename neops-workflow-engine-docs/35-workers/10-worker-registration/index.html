    Worker Registration

    Workers in the Neops Workflow Engine are processes that execute function blocks. The worker registration system allows workers to:

    • Register themselves with the workflow engine
    • Send heartbeat pings to indicate they are online
    • Associate function blocks they can execute
    • Be tracked for online/offline status

    Data Model

    WorkerRegistration Entity

    Field Type Description
    uuid UUID Unique identifier (auto-generated)
    version string Worker software version (required)
    name string Human-readable label (optional)
    ipAddress string IP address of the worker (optional)
    lastPingAt Date Timestamp of last heartbeat
    createdAt Date When the worker was registered
    updatedAt Date When the worker was last updated
    deletedAt Date When the worker was soft-deleted (null if active)

    Worker-FunctionBlock Relationship

    Workers have a many-to-many relationship with FunctionBlockRegistrations: - One worker can have multiple function blocks - One function block can be available on multiple workers

    Configuration

    The following environment variables control worker behavior:

    Variable Default Description
    WORKER_UNREACHABLE_THRESHOLD_MS 120000 (2 min) Time without ping before worker is considered unreachable
    WORKER_OFFLINE_THRESHOLD_MS 360000 (6 min) Time without ping before worker is considered offline (default: 3x unreachable)
    WORKER_OFFLINE_DELETE_THRESHOLD_MS 86400000 (1 day) Time before offline workers are auto-deleted by periodic cleanup
    WORKER_CLEANUP_ENABLED true Enable/disable automatic cleanup
    WORKER_CLEANUP_INTERVAL_MS 3600000 (1 hour) How often to run the cleanup job
    BLACKBOARD_JOB_CHECK_INTERVAL 60000 (1 min) How often to check for stuck jobs
    BLACKBOARD_STUCK_JOB_TIMEOUT Computed (default: 2x offline threshold = 12 min) Time before stuck POLLED jobs are failed

    Note: BLACKBOARD_STUCK_JOB_TIMEOUT defaults to 2 * WORKER_OFFLINE_THRESHOLD_MS (12 minutes with default settings). If you customize WORKER_OFFLINE_THRESHOLD_MS, this value adjusts accordingly.

    Manual Cleanup (Offline Workers > 1 day)

    Removes workers that have been offline longer than WORKER_OFFLINE_DELETE_THRESHOLD_MS (default: 1 day). 

    POST /workers/cleanup

    Response: 

    {
  "cleanedUpWorkerIds": ["uuid-1", "uuid-2", "uuid-3"],
  "failedJobCount": 5
}

    Manual Cleanup (All Offline Workers - Immediate)

    Removes all currently offline workers (offline > WORKER_OFFLINE_THRESHOLD_MS, default: 6 min). 

    POST /workers/cleanup-offline

    Response: 

    {
  "cleanedUpWorkerIds": ["uuid-1", "uuid-2"],
  "failedJobCount": 3
}

    Function Block Registration with Worker

    To associate a function block with a worker, include the optional workerId field when registering:

    POST /function-blocks/register
Content-Type: application/json

{
  "package": "network",
  "name": "ping",
  "majorVersion": 1,
  "minorVersion": 0,
  "patchVersion": 0,
  "description": "Ping a host",
  "fbType": "execute",
  "parameterJsonSchema": { ... },
  "resultDataJsonSchema": { ... },
  "workerId": "123e4567-e89b-12d3-a456-426614174000"  // Optional: links FB to worker
}

    Get Workers for Function Block

    GET /function-blocks/{pkg}/{name}/{version}/workers

    Worker Lifecycle

    Registration Flow

    1. Worker starts up
    2. Worker calls POST /workers/register with version info
    3. Worker receives UUID to use for subsequent calls
    4. Worker stores UUID for the session

    Heartbeat Loop

    1. Worker sets up a periodic timer (recommended: every 30 seconds)
    2. Worker calls POST /workers/{uuid}/ping
    3. If ping returns 404, worker re-registers (UUID was cleaned up)

    Function Block Registration

    1. After receiving UUID, worker registers its function blocks
    2. Each function block registration includes the workerId
    3. The workflow engine associates the function block with the worker

    Worker States

    A worker can be in one of three states:

    ONLINE: Worker is actively pinging and available for jobs - lastPingAt is within WORKER_UNREACHABLE_THRESHOLD_MS (default: 2 minutes)

    UNREACHABLE: Worker has not pinged recently but may recover - lastPingAt is older than WORKER_UNREACHABLE_THRESHOLD_MS but within WORKER_OFFLINE_THRESHOLD_MS

    OFFLINE: Worker has been unresponsive for an extended period - lastPingAt is older than WORKER_OFFLINE_THRESHOLD_MS (default: 6 minutes) - Jobs assigned to offline workers will be automatically failed by the stuck job cleanup

    Cleanup Lifecycle

    The workflow engine runs two periodic cleanup services:

    Job Cleanup Service (runs every 1 minute)

    • Stuck Job Cleanup: Fails jobs stuck in POLLED state longer than BLACKBOARD_STUCK_JOB_TIMEOUT
    • This catches all stuck jobs, including those from offline workers

    Worker Cleanup Service (runs every 1 hour)

    • Offline Worker Cleanup: Soft-deletes workers offline > WORKER_OFFLINE_DELETE_THRESHOLD_MS (1 day)
    • Associated jobs are failed before each worker is deleted
    Timeline of a worker going offline:

0 min      - Worker stops pinging
2 min      - Worker state: UNREACHABLE
6 min      - Worker state: OFFLINE
12 min     - Stuck jobs for this worker are failed (stuck job timeout)
1 day      - Worker is auto-deleted (offline delete threshold)

    Graceful Shutdown

    Workers can gracefully unregister themselves:

    // On shutdown
await fetch(`http://workflow-engine/workers/${uuid}/unregister`, {
  method: 'POST'
});

    This immediately sets the worker to OFFLINE state without deleting it. The worker can come back online later by sending a ping.

    Error Handling

    Scenario HTTP Status Description
    Worker not found 404 UUID doesn't exist or was cleaned up
    Missing required field 400 Version is required for registration
    Invalid UUID format 400 UUID parameter is not valid

    Example: Complete Worker Integration

    // Worker startup
async function initWorker() {
  // 1. Register with workflow engine
  const response = await fetch('http://workflow-engine/workers/register', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
      version: '1.0.0',
      name: 'my-worker',
      ipAddress: getLocalIp()
    })
  });
  const { uuid } = await response.json();

  // 2. Register function blocks
  for (const fb of myFunctionBlocks) {
    await fetch('http://workflow-engine/function-blocks/register', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({
        ...fb,
        workerId: uuid
      })
    });
  }

  // 3. Start heartbeat loop
  setInterval(async () => {
    const pingResponse = await fetch(`http://workflow-engine/workers/${uuid}/ping`, {
      method: 'POST'
    });

    if (pingResponse.status === 404) {
      // UUID was cleaned up, re-register
      initWorker();
    }
  }, 30000); // Every 30 seconds
}

// Graceful shutdown
process.on('SIGTERM', async () => {
  await fetch(`http://workflow-engine/workers/${uuid}/unregister`, {
    method: 'POST'
  });
  process.exit(0);
});
