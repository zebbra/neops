    Testing framework for FunctionBlock tests (Netlab‑backed)

    This framework is for beein able to test our neops‑worker‑sdk FunctionBlocks easily with two simple decorators.

    It also provides an easy abstraction to spin up virtual notwork labs using Netlab ** (netlab.tools). Those labs can run locally (on Ubuntu machines), on a remote VM which is Running the Remote Lab Manager** or executed in CI on our custom GitHub runner.

    Who is this for: engineers writing/maintaining FunctionBlocks who want quick, repeatable tests against realistic router labs.

    What you get

    You get a predictable test harness:

    • Easy to use decorators for testing function blocks
    • Lab reuse to speed up suites / test state changes over multiple functionblocks (WIP)
    • Deterministic grouping/order for tests sharing a lab fixture
    • Zero‑config remote: set one env var and fixtures switch to the Remote Lab

    Before you start

    When using remote labs, Networking comes first. Without reachability to the lab subnet(s), nothing else will work.

    Networking via Headscale/Headplane (Tailscale clients)

    We self‑host the Tailscale control plane using Headscale and Headplane. Follow the dedicated guide to deploy the stack with Docker Compose and connect both the Remote Lab host and your peers:

    • See: Headscale + Headplane with Docker Compose

    Key host settings (unchanged):

    # Docker must not manage iptables when routing to the lab networks
sudo mkdir -p /etc/docker
echo '{"iptables": false}' | sudo tee /etc/docker/daemon.json
sudo systemctl restart docker || true

# Enable IP forwarding
sudo sysctl -w net.ipv4.ip_forward=1
sudo sed -i '' 's/^#\?net.ipv4.ip_forward.*/net.ipv4.ip_forward = 1/' /etc/sysctl.conf
sudo sysctl -p || true

    WireGuard (WIP)

    We plan to substitute Tailscale with WireGuard. Settings mirror Tailscale (Docker iptables disabled, IP forwarding enabled). For the WIP guide see: WireGuard configuration. Official docs: wireguard.com.

    Do I need Netlab locally?

    Only if you want to run labs on your machine (Ubuntu only) or when preparing a remote Ubuntu VM to host labs.

    For purely remote runs (using REMOTE_LAB_URL), your local machine does not need Netlab or Containerlab.

    If you do need Netlab locally or on a remote Ubuntu VM, follow the Netlab installation guide. External docs: Netlab ** (netlab.tools), Containerlab** (containerlab.dev).

    Python dependencies

    # Local testing dependencies
uv sync --extra test

# Remote Lab server dependencies (on the server host)
uv sync --extra test

    Use labs as fixtures in FunctionBlock tests

    • Local: no REMOTE_LAB_URL → labs run locally via Netlab/Containerlab.
    • Remote (recommended): set REMOTE_LAB_URL=http://<remote_host>:8000 → tests acquire labs via the Remote Lab API.

    Define fixtures with create_netlab_fixture(...). Fixtures yield NetlabDevice objects representing lab nodes. Use them inside FunctionBlock tests via the decorators below.

    # tests/conftest.py
from neops_worker_sdk.testing.netlab.fixture import create_netlab_fixture

simple_frr = create_netlab_fixture("tests/topologies/simple_frr.yml")
simple_iol = create_netlab_fixture("tests/topologies/simple_iol.yml", reuse_lab=True)

    Notes

    • reuse_lab=True lets multiple tests share the same topology instance.
    • Each test may use at most one Netlab fixture (enforced by the ordering plugin) since only one netlab instance can run at once.

    Decorators for FunctionBlocks

    The two styles:

    • Fixture‑powered (recommended): execute_fb_test_case_with_fixtures(...)
    • Direct context (no fixture): execute_fb_test_case_with(...) — use for physical devices on the network. This decorator is likely to be removed; prefer fixture‑powered tests.

    The fixture‑powered decorator uses the referenced fixture, converts its NetlabDevice objects to the neops‑workflow‑engine’s DeviceTypeDto via a callback, builds a WorkflowContext, and runs your FunctionBlock with @pytest.mark.asyncio.

    Execution order and grouping:

    Tests using the same fixture are grouped to run consecutively; original order within the group is preserved. Only one Netlab fixture per test is allowed.

    Full Reference of @execute_fb_test_case_with_fixtures

    Signature (simplified):

    def fb_test_case_with_lab(
        test_description: str,
        params: P,
        base_devices: list[DeviceTypeDto] | None = None,
        base_device_groups: list[DeviceGroupTypeDto] | None = None,
        *,
        fixture: str | None = None,
        succeeds: bool = True,
        expected_result_data: R | None = None,
        assertions: list[Callable[[FunctionBlockResult[R]], bool]] | None = None,
) -> Callable[[type[FunctionBlock[P, R]]], type[FunctionBlock[P, R]]]:
    ...

    Parameters:

    • test_description: readable description; becomes part of the generated test name
    • params: your FunctionBlock’s parameter object
    • base_devices: optional static devices (DeviceTypeDto) always present in context
    • base_device_groups: optional groups
    • fixture: name of a pytest fixture created via create_netlab_fixture(...) in conftest.py
    • fixture_to_device: converter from BaseMockDevice (e.g., NetlabDevice) to DeviceTypeDto (required when fixture is used)
    • succeeds: expected success flag of the FunctionBlock result
    • expected_result_data: optional strict equality check against FunctionBlock data
    • assertions: additional callables receiving the FunctionBlockResult

    Behavior:

    • Generates an async pytest test function named test_<class>_<case_id>
    • Adds the referenced fixture as a function argument so pytest injects its value - our Netlab fixture abstraction will either spin up the lab locally, on remote or reuse it if it was already running
    • Builds a WorkflowContext from base_devices + converted fixture devices
    • Executes your FunctionBlock and applies expectations/assertions

    Examples

    # tests/function_blocks/fb_ping.py
@fb_test_case_with_lab(
    "Ping FRR device",
    RandomPingParameters(ping_amount=2),
    fixture="simple_frr",
)

    # tests/function_blocks/fb_get_show_cmd.py
@fb_test_case_with_lab(
    "Show version with Netlab IOL fixture",
    ShowCmdParameters(cmd="show version", password="admin"),
    fixture="simple_iol",
    assertions=[
        lambda r: r.data is not None,
        lambda r: r.data.output is not None,
        lambda r: r.success is True,
        lambda r: "Cisco" in r.data.output,
    ],
)

    Full Reference of @execute_fb_test_case_with (...)

    Signature (simplified):

    def execute_fb_test_case_with(
        test_description: str,
        params: P,
        context: WorkflowContext,
        *,
        succeeds: bool = True,
        expected_result_data: R | None = None,
        assertions: list[Callable[[FunctionBlockResult[R]], bool]] | None = None,
) -> Callable[[type[FunctionBlock[P, R]]], type[FunctionBlock[P, R]]]:
    ...

    Usage notes:

    • Use this when testing against physical devices reachable on the network by constructing a WorkflowContext yourself
    • This decorator is likely to be removed in favor of fixture‑powered tests; prefer execute_fb_test_case_with_fixtures

    Examples:

    This tests against a device available in the local network of the machine running the test on.

    # tests/function_blocks/fb_get_show_cmd.py
@execute_fb_test_case_with(
    "Show version with local network device",
    ShowCmdParameters(cmd="show version", password="PASSWORD"),
    create_workflow_context([DeviceTypeDto(hostname="zbr-chbrn-was-csw-202", ip="10.0.200.11", id="device-1")]),
    assertions=[
        lambda r: r.data is not None,
        lambda r: r.data.output is not None,
        lambda r: r.success is True,
        lambda r: "Cisco" in r.data.output],  # type: ignore
)

    Remote Lab usage (Hetzner VM)

    Start the server on the remote host:

    uv sync --extra test
uv run remote_lab --host 0.0.0.0 --port 8000 --log-level info

    Point tests at the Hetzner VM:

    export REMOTE_LAB_URL=http://91.99.184.46:8000
pytest -m testing

    More on endpoints and behavior: Remote Lab Manager.

    Troubleshooting

    • Containerlab ≥ 0.63; containerlab version must work without sudo prompts.
    • For FRR labs on Ubuntu, ensure the VRF kernel module is installed and loaded (see Netlab configuration).
    • Remote server may respond with 423 Locked while the lab is busy; your session will be promoted in FIFO order.
    • For local routing to lab subnets, disable Docker iptables and enable IP forwarding as shown in Tailscale/WireGuard sections.
    • On clients, ensure Tailscale routes are accepted and the subnet route is visible in ip route.
